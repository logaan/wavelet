# Standard library

`wavelet:std/core` is implicitly imported and open in every file. Because identifiers are WIT labels, there is no `+` or `<` — the names are short and boring, a deliberate cost of "the syntax is WAVE, no exceptions".

Every function below is **implemented and runnable today**. A call bundles its arguments — `add(2 3)` passes two — so most functions read several arguments positionally; a few take a single value. Edit any example and press Run.

:::note[What "function" means here]
These are the core *builtins*. Convenience **macros** (`And`, `Or`, `TryLet`) are **not** shipped — define them yourself with [`DefMacro`](./language/macros). Two pre-bound *values* also live in core: `none` (the empty option) and `pi`.
:::

## Input / output

The standard library has **no** I/O builtins. The language and compiler know
nothing about WASI: a program that wants to write to stdout, read its arguments,
or talk to the host imports the relevant WIT interface (e.g. `wasi:cli/stdout`,
`wasi:cli/environment`, `wasi:io/streams`) and calls it like any other
[imported component](./components). The `cli` template scaffolded by
[`wavelet new`](./cli#new) shows the pattern: it imports `wasi:cli/stdout` and
`wasi:io/streams`, builds the greeting, and writes it through the stream.

Because these are host calls, they only run in a compiled component on a wasm
runtime — not in the in-browser playground or under [`wavelet run`](./cli#run),
which evaluate pure programs only.

## Cells

A `cell` is the standard library's mutable reference — the one place mutation lives, since [the core has none](./language/evaluation#no-loops-no-mutation-no-early-return).

| Function | Shape | Effect |
|---|---|---|
| `cell-new` | `cell-new(v)` | a new cell holding `v` |
| `cell-get` | `cell-get(c)` | read the current value |
| `cell-set` | `cell-set(c v)` | overwrite the value (returns unit) |
| `drop` | `drop(h)` | release a resource handle |

<Playground id="std-cells" />

:::info[Resources beyond cells]
`cell` is the only resource type that ships today. General resource handles — opening, methods, owned-handle drop semantics — are part of the component-interop story and are **not yet implemented**; `drop` currently accepts any value and returns unit. See [Components & composition](./components#resources) and the [roadmap](./roadmap).
:::
