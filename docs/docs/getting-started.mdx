# Getting started

## The fastest way to try it

You don't need to install anything to experiment — the playgrounds throughout these docs run a real Wavelet interpreter in your browser. Start with the [standard library](./stdlib) or the [special forms](./language/special-forms) and edit the examples.

<Playground id="gs-hello" autoRun />

To run Wavelet on real components — to build `.wasm`, compose, and execute on a host like [wasmtime](https://wasmtime.dev/) — install the compiler.

## External tools

Building and composing components shells out to two [Bytecode Alliance](https://bytecodealliance.org/) CLIs, so they must be on your `PATH`:

- [**`wkg`**](https://github.com/bytecodealliance/wasm-pkg-tools) fetches dependency WIT (e.g. `wasi:cli`, `wasi:http`) into a project's `wit/` tree and maintains a `wkg.lock` lock file.
- [**`wac`**](https://github.com/bytecodealliance/wac) composes the project's components into a single artifact.

The Homebrew formula declares both as dependencies, so `brew install` brings them along. Installing by hand: `cargo install wkg wac-cli`. Neither is needed for `wavelet run` or the in-browser interpreter, which evaluate pure programs and don't emit components.

## Installing with Homebrew

On macOS and Linux the quickest way to get the `wavelet` CLI and the `wavelet-lsp` language server is the personal [tap](https://github.com/logaan/homebrew-tap):

```console
$ brew install logaan/tap/wavelet
```

This puts both `wavelet` and `wavelet-lsp` on your `PATH` as prebuilt binaries — no Rust toolchain is fetched. To track `main` instead of the latest release (built from source), use `brew install --HEAD logaan/tap/wavelet`.

## Building the compiler

Prefer to build it yourself? Wavelet is written in Rust (2024 edition).

```console
$ git clone https://github.com/logaan/wavelet
$ cd wavelet
$ cargo build            # debug binary at ./target/debug/wavelet
$ cargo build --release  # optimized binary at ./target/release/wavelet
$ cargo test             # run the test suite
```

Add `./target/release` to your `PATH`, or invoke the binary directly.

## Editor support

`.wvl` files get syntax highlighting plus the `wavelet-lsp` language server (diagnostics, completion, hover, document symbols). The highlighting grammars are derived from the compiler's lexer, so they match the language exactly. `wavelet-lsp` ships as a standalone `wavelet-lsp-<platform>` binary on the [releases page](https://github.com/logaan/wavelet/releases/latest) and is used automatically by the VS Code extension and the Neovim plugin.

### Neovim (LazyVim / lazy.nvim)

The Neovim plugin lives in its own repo, [`logaan/wavelet.nvim`](https://github.com/logaan/wavelet.nvim). Add it to LazyVim with a spec in `~/.config/nvim/lua/plugins/wavelet.lua`:

```lua
return {
  {
    "logaan/wavelet.nvim",
    ft = "wavelet",
    init = function()
      vim.filetype.add({ extension = { wvl = "wavelet" } })
    end,
  },
}
```

Open any `.wvl` file and it is highlighted. For language features, put the `wavelet-lsp` server on your `PATH` (`brew install logaan/tap/wavelet`, `cargo install --path tooling/wavelet-lsp`, or download a prebuilt binary from the releases page) — the plugin starts it automatically. Point at a specific binary with `vim.g.wavelet_lsp_path`.

### VS Code

Download `wavelet-vscode.zip`, unzip it into your extensions folder, and reload the window:

```console
$ curl -L -o wavelet-vscode.zip \
    https://github.com/logaan/wavelet/releases/latest/download/wavelet-vscode.zip
$ unzip wavelet-vscode.zip -d ~/.vscode/extensions/
```

The extension is self-contained — it bundles the matching `wavelet-lsp` server, so language features work with no extra download. Build-from-source and plugin-manager alternatives are documented in [`tooling/`](https://github.com/logaan/wavelet/tree/main/tooling).

## Hello, world

The quickest way to a runnable component is [`wavelet new`](./cli#new), which scaffolds a project, fetches its dependency WIT with `wkg`, and lays down build/run scripts:

```console
$ wavelet new hello
created hello/
  hello/.gitignore
  hello/README.md
  hello/src/greeting.wvl
  hello/src/main.wvl
  hello/scripts/build.sh
  hello/scripts/run.sh

$ cd hello
$ scripts/run.sh           # build, then run with wasmtime
Hello, world!
$ scripts/run.sh Ada
Hello, Ada!
```

The default `cli` template lays down two components. `src/greeting.wvl` is the pure domain model:

```wavelet
Package "hello:greeting@0.1.0"

Export greet
Def greet Fn {name: string}
  str-cat["Hello, " name "!"]
```

`src/main.wvl` is the entry point. It adopts the CLI world by **exporting `wasi:cli/run`** — there is no compiler-known target world — and does its output and argument reading through ordinary imported WASI interfaces (`wasi:cli/stdout`, `wasi:io/streams`, `wasi:cli/environment`), lowered through the generic WIT bridge:

```wavelet
Package "hello:main@0.1.0"

Import {pkg: "hello:greeting/api" as: greeting}
Import {pkg: "wasi:cli/stdout@0.2.0" as: stdout}
Import {pkg: "wasi:io/streams@0.2.0" as: streams}

Export {iface: "wasi:cli/run" name: run result: result}
Def run Fn {}
  Let {out: stdout/get-stdout()}
    Do [streams/blocking-write-and-flush[out
          str-cat[greeting/greet("world") "\n"]]
        ok(0)]
```

## Project layout

A scaffolded project pins its dependency WIT in the repository:

```text
hello/
  src/            # your .wvl components
  wit/            # synthesized world + fetched dependency WIT
    deps/         # wkg-fetched packages (wasi:cli, wasi:io, …)
  wkg.lock        # pinned dependency versions
  out/            # build artifacts (git-ignored)
```

`wavelet build` synthesizes each component's world into `wit/`, ensures the dependency WIT is present (via `wkg`), emits the components, and composes them with `wac` into a single `out/app.wasm`.

## Run vs. build

Wavelet has two execution paths, and understanding the split is the key to the workflow:

| | `wavelet run` | `wavelet build` |
|---|---|---|
| What it does | Interprets the files directly | Emits real `.wasm` components and composes them |
| Speed | Instant — no codegen | Slower; produces a deployable artifact |
| Output | The value of the program | One `out/app.wasm` you can ship and run on any host |
| Role | The semantics oracle; fastest iteration | Production artifacts |

The interpreter exists so the language's semantics can be validated independently of code generation — the wasm backend is checked against it. It evaluates **pure** programs: calls into host interfaces (`wasi:*`) only run in a built component on a wasm runtime, so `run` is for the logic, `build` for anything that touches the world.

## A two-component program

The `greeting` and `main` files above are a two-component program: `main.wvl` imports `hello:greeting/api`, and `greeting.wvl` declares `Package "hello:greeting@…"`, so the composer wires them together. `wavelet build` emits each component and links them into one artifact:

```console
$ wavelet build src/*.wvl -o out
built out/app.wasm
$ wasmtime run out/app.wasm Ada
Hello, Ada!
```

You can compose hand-picked components yourself with [`wavelet compose`](./cli#compose) instead; `wavelet build` runs it for you.

See the [CLI reference](./cli) for every command and flag, and [Components & composition](./components) for how files become components and link together.
