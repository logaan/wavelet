# CLI reference

The `wavelet` binary is built with `cargo build` (see [Getting started](./getting-started#building-the-compiler)). It exposes eight subcommands: `new` scaffolds a project, and the rest are organised around the compiler pipeline **read → expand → analyze → emit → componentize**.

```text
wavelet new <name> [--type=cli|http]                        # scaffold a new project
wavelet read [file.wvl]                                      # parse → canonical WAVE tree (stdin if no file)
wavelet expand <file.wvl>                                    # run macros to fixpoint
wavelet wit <file.wvl>                                       # synthesized WIT world
wavelet repl                                                 # interactive read-eval-print loop
wavelet run <file.wvl>... [-- <args>...]                     # interpret directly (no codegen)
wavelet build <file.wvl>... [-o <dir>]                       # compile each file to a .wasm component
wavelet compose <entry.wasm> <plug.wasm>... [-o <app.wasm>]  # link components (auto-plug)
```

---

## `new`

Scaffold a fresh project into a new directory of the given name. Every template
lays down a `.gitignore`, a `src/` with an entry point and the domain model it
imports across the component boundary, build/run scripts, and a short README.
`--type` chooses the template; `cli` is the default.

### `--type=cli` (default)

A command-line program: `src/main.wvl` adopts the CLI world by exporting
[`wasi:cli/run`](./language/special-forms#export) and greets its first argument,
delegating the wording to the pure `greet` function in `src/greeting.wvl`. Its
output and argument reading go through imported WASI interfaces
(`wasi:cli/stdout`, `wasi:io/streams`, `wasi:cli/environment`), whose WIT is
fetched into `wit/` by `wkg`.

```console
$ wavelet new my-app
created my-app/
  my-app/.gitignore
  my-app/README.md
  my-app/src/greeting.wvl
  my-app/src/main.wvl
  my-app/scripts/build.sh
  my-app/scripts/run.sh

$ cd my-app
$ scripts/run.sh Ada          # build, then run with wasmtime
Hello, Ada!
```

`new` also runs `wkg wit fetch` to populate the project's `wit/deps/` with the
dependency WIT and write a `wkg.lock`, so a fresh project has its WASI
dependencies pinned. `scripts/build.sh` compiles `src/*.wvl` into a git-ignored
`out/` and composes them into `out/app.wasm`; `scripts/run.sh` does that and then
runs it with `wasmtime`, passing through any extra arguments.

### `--type=http`

A web app whose front end, `src/app.wvl`, implements the
[`wasi:http/incoming-handler`](https://github.com/WebAssembly/wasi-http)
interface and renders a stateless HTML page: it greets via `src/greeting.wvl`
(the pure `greet` function, imported across the component boundary) and echoes
the request path. `scripts/serve.sh` serves the component with `wasmtime serve`.
The front end needs no special language support — it imports `wasi:http/types`
and `wasi:io/streams` (their WIT fetched into `wit/` by `wkg`) and drives the
response pipeline with ordinary calls into those interfaces, the same way any
component implements any WIT interface.

## `read`

Parse a file and print the **canonical WAVE form tree** — every piece of [reader sugar](./language/syntax#desugaring) resolved. With no path argument it reads from stdin.

```console
$ wavelet read shout.wvl
$ echo 'shout("hi")' | wavelet read
```

Use this to see exactly how the attachment rule, qualified calls, and TitleCase macro sugar desugar. It is the ground truth for "what does this syntax mean as data".

## `expand`

Run macros to fixpoint and print the resulting forms. `DefMacro` definitions are evaluated (registering the macro) and dropped; every macro use is rewritten to its expansion.

```console
$ wavelet expand program.wvl
```

This is `read` plus the [macro expander](./language/macros) — what the analyzer and emitter actually see.

## `wit`

Print the **WIT world** synthesized from a file's typed `Fn` params, `Export` records, and `DefType`s. Doc comments (`///`) are surfaced as WIT docs.

```console
$ wavelet wit shout.wvl
package demo:shout@0.1.0;

interface api {
  shout: func(phrase: string) -> string;
}

world shout {
  export api;
}
```

## `repl`

An interactive read-eval-print loop. Enter forms; multi-line input continues until a form is complete; `Ctrl-D` exits. Because values print as WAVE and code *is* WAVE, the REPL's output is always valid input.

```console
$ wavelet repl
wavelet repl — enter forms, Ctrl-D to exit
> add(2 3)
5
> map(Fn {x} mul(x x) [1 2 3])
[1, 4, 9]
```

:::tip[This is what the playground runs]
The [in-browser playgrounds](./intro.mdx#try-it) throughout these docs are this same evaluator, compiled to WebAssembly. Anything that works in a playground works in `repl` and `run`.
:::

## `run`

Interpret a set of files together — **no code generation**. Resolves `Import`s by package id, honours `Export`/`as:`/`open:`, and calls the entry file's exported `run`. The fastest way to exercise a program's logic.

```console
$ wavelet run main.wvl shout.wvl
```

The interpreter evaluates **pure** programs: it has no host interfaces, so a `run` that calls into `wasi:*` only works in a built component on a wasm runtime. Use `run` to validate cross-component logic; build the component for anything that touches the world. The first file listed is the entry component. (Anything after `--` is still accepted and ignored, for compatibility with the build flow.)

## `build`

Compile each file to a real `.wasm` component — core wasm (linear-memory value boxes, `return_call` tail calls) wrapped with canonical-ABI lift/lower and componentized via `wasm-tools` — **and compose them into a single `out/app.wasm`**. Each cross-component import is wired to the sibling that exports it via a generated `out/app.wac` run through `wac`; host imports (`wasi:*`) are left unsatisfied for the runtime. Output goes to `out/` by default, or `-o <dir>`.

```console
$ wavelet build src/*.wvl -o out
built out/app.wasm
$ wasmtime run out/app.wasm
```

The individual `out/<package>.wasm` components are left alongside `app.wasm`. Dependency WIT must already be present in `wit/deps/` (run `wavelet new`, or `wkg wit fetch`, first).

## `compose`

Compose hand-picked components by **auto-plugging**: each unresolved import is matched by package id (and version compatibility) against the exports of the other components on the command line, in the spirit of `wac plug`. `wavelet build` does this for you; `compose` is the manual alternative. The first argument is the entry component. Output is `app.wasm` by default, or `-o <file>`.

```console
$ wavelet compose out/demo-main.wasm out/demo-shout.wasm -o app.wasm
app.wasm
```

Because this drives standard `wasm-tools` / WAC machinery, the result participates in any component ecosystem workflow.

:::info[Not yet implemented]
A few CLI features described in the language design are still on the [roadmap](./roadmap): a `wavelet add <pkg>` command (fetch WIT from a registry and pin a version), a WAVE compose **manifest** for resolving ambiguities and host substitutions, and `compose --fuse` for merging all-Wavelet subgraphs.
:::
