# Components & composition

In Wavelet, **one file is one component**. This page covers how a `.wvl` file declares its place in the component world, and how `build` and `compose` turn files into linked, runnable WebAssembly.

## Anatomy of a file

A file is a flat sequence of forms: an optional [`Package`](./language/special-forms#package), any number of [`Import`](./language/special-forms#import)s, then [`DefType`](./language/special-forms#deftype)s, [`Def`](./language/special-forms#def)s, and [`Export`](./language/special-forms#export)s in any order.

```wavelet
// shout.wvl
Package "demo:shout@0.1.0"

Export shout
Def shout Fn {phrase: string}
  str-cat(upper(phrase) "!")
```

The compiler synthesizes a WIT world from these forms. You can see it with [`wavelet wit`](./cli#wit):

```wit
package demo:shout@0.1.0;

interface api {
  shout: func(phrase: string) -> string;
}

world shout {
  export api;
}
```

Exported functions get their WIT signatures from typed `Fn` parameters, from inference against use, or from an explicit record form. A file's exports land in a default interface named `api`; an `Export` record with an `iface` field groups them otherwise.

## Packages

[`Package`](./language/special-forms#package) declares the component's id and version:

```wavelet
Package "demo:shout@0.1.0"
```

Every file implicitly imports the standard library (`wavelet:std/core`, open).

## Imports

[`Import`](./language/special-forms#import) takes a package path string, or a record for control:

```wavelet
Import "wasi:cli/environment"                       // alias defaults to last segment
Import {pkg: "wasi:keyvalue/store@0.2.0" as: kv}     // explicit alias
Import {pkg: "demo:text/api" open: true}             // splat names in unqualified
```

Imported names are used qualified: `kv/open`, `kv/get`. With `open: true`, the names are also bound unqualified.

When you run a set of files together with [`wavelet run`](./cli#run), imports are resolved **by package id** against the other files on the command line — an interpreter stand-in for the composer:

```console
$ wavelet run main.wvl shout.wvl
```

`main.wvl` imports `demo:shout/api`; because `shout.wvl` declares `Package "demo:shout@…"`, the runner wires them together and calls `main`'s exported `run`. The interpreter evaluates **pure** logic only — calls into host interfaces (`wasi:*`) run only in a built component on a wasm runtime — so `run` is for exercising your cross-component logic, while output and side effects happen under `wavelet build` + a runtime.

## Exports

[`Export`](./language/special-forms#export) publishes a definition. A bare name exports into the default `api` interface:

```wavelet
Export shout
```

A record form gives finer control over the signature and interface grouping:

```wavelet
Export {name: shout params: {phrase: string} result: string}
Export {name: render iface: "html"}
```

### Adopting a host world

There is no compiler-known "target world". A component **implements** an external interface by exporting it: name the interface with the `iface` field and the compiler emits the export under that interface's WIT signature. A CLI program exports `wasi:cli/run`; an HTTP front end exports `wasi:http/incoming-handler`.

```wavelet
Export {iface: "wasi:cli/run" name: run result: result}
```

The interface's WIT comes from `wit/deps/` (fetched by `wkg`, see [below](#project-layout-and-external-wit)), so the compiler treats `wasi:cli/run` exactly as it would any third-party interface.

## Resources

Resources come along for free in the design, because WIT canonicalizes methods as functions whose first parameter is the handle: `kv/open({name: "default"})` returns a handle, and `kv/get(bucket "greeting")` calls a method on it. Owned handles are dropped when their binding scope ends; `drop(h)` forces it.

## Project layout and external WIT

A Wavelet project carries a `wit/` tree alongside its `src/`:

```text
my-app/
  src/            # your .wvl components
  wit/            # synthesized world + fetched dependency WIT
    deps/         # wkg-fetched packages (wasi:cli, wasi:http, …)
  wkg.lock        # pinned dependency versions
  out/            # build artifacts (git-ignored)
```

The compiler vendors **no** WASI WIT of its own. Any host interface a component imports or exports (`wasi:cli/stdout`, `wasi:http/types`, a third party's `acme:foo/bar`) is fetched as ordinary WIT into `wit/deps/` by [**`wkg`**](https://github.com/bytecodealliance/wasm-pkg-tools) and parsed at build time, with versions pinned in `wkg.lock`. [`wavelet new`](./cli#new) fetches them when it scaffolds; `wavelet build` synthesizes each component's world into `wit/` and ensures the deps are present before emitting. So WASI is "just some WIT packages you import," resolved through the same mechanism as your own components.

## Composition

[`wavelet build`](./cli#build) turns each file into a component **and composes them into one artifact**, `out/app.wasm`, wiring every cross-component import to the sibling that exports it via a generated `out/app.wac` run through [**`wac`**](https://github.com/bytecodealliance/wac). Host imports (`wasi:*`) are left unsatisfied for the runtime to provide.

```console
$ wavelet build src/*.wvl -o out
built out/app.wasm
$ wasmtime run out/app.wasm wasm
WASM!
```

The individual `out/<package>.wasm` components are left alongside `app.wasm`. To compose hand-picked components yourself, [`wavelet compose`](./cli#compose) auto-plugs them: every unresolved import is matched by package id against the exports on the command line, in the spirit of `wac plug`.

```console
$ wavelet compose out/demo-main.wasm out/demo-shout.wasm -o app.wasm
app.wasm
```

Boundary calls copy data (the canonical ABI).

:::info[Status]
`build` (composing via `wac`), `compose` (auto-plug), and `wit` synthesis are implemented. A WAVE-document compose **manifest** for settling ambiguities/substitutions is **not yet implemented**. See the [roadmap](./roadmap).
:::

## A three-language program

The point of the model: `main` can't tell which of its imports is foreign.

```wit
// the Rust side exports this WIT
package acme:words@1.0.0;
interface picker { pick: func(seed: u64) -> result<string, string>; }
```

```wavelet
// main.wvl
Package "demo:main@0.1.0"

Import {pkg: "acme:words/picker@1.0.0" as: words}
Import {pkg: "demo:caps/api"           as: caps}

Export pick-word
Def pick-word Fn {} -> string
  Match words/pick({seed: 42}) [
    (ok(w)  caps/shout({phrase: w}))
    (err(e) str-cat("no word today: " e))
  ]
```

`main` calls `words/pick` (the Rust component) and `caps/shout` (a Wavelet sibling) identically — neither call reveals which language is on the other side. Build composes the Wavelet components; a third-party component is wired in with `wavelet compose`:

```console
$ cargo component build --release          # acme_words.wasm
$ wavelet build src/*.wvl -o out
$ wavelet compose out/app.wasm target/release/acme_words.wasm -o final.wasm
```
