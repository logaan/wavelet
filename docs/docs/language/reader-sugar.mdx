# Reader sugar

Wavelet's tokens *are* [WAVE](https://github.com/bytecodealliance/wasm-tools/tree/main/crates/wasm-wave)'s tokens, and the reader is where all sugar dies: every surface form desugars to a canonical WAVE value before anything is evaluated. This page enumerates each reader-sugar form, one per section, with the canonical value it produces.

A form is **reader sugar** when it is *not already* a valid WAVE value, or when the reader **transforms** it into a different shape as it reads. That is the whole list below.

You can confirm any desugaring yourself with `wavelet read`, which prints the canonical WAVE tree for a piece of source:

```console
$ echo 'f(x y)' | wavelet read
(f, x, y)
```

## 1. Commas are whitespace

Commas are whitespace. These read identically:

```wavelet
[1 2 3]
[1, 2, 3]
```

```console
$ echo '[1 2 3]' | wavelet read
[1, 2, 3]
```

This is the sugar that makes Wavelet a *superset* of WAVE rather than a different language: drop the commas and a WAVE value is already Wavelet. The canonical printer always puts the commas back, so anything Wavelet reads prints as strict WAVE and can be handed to any WAVE tooling. Because every comma-free tuple, list, record, and flag set is just this rule applied, none of them needs its own entry here.

## 2. The attachment rule (call forms)

Wavelet's only whitespace-sensitive rules are about **attachment** — a token abutting the one before it. The first is the call rule: a `(` that *immediately* follows an identifier — with no intervening whitespace — makes that identifier the **head** of a call, and the reader folds the head and the parenthesized forms into a single tuple. With whitespace in between, the parenthesized expression is a separate, free-standing value:

```wavelet
delete-file({path: "foo.md" force: true})   // a call: the head plus a record argument
delete-file ({path: "foo.md" force: true})  // two forms: a variable, then a record in a tuple
```

A call desugars to a **tuple whose first element is the head**:

| You write | Canonical WAVE form | |
|---|---|---|
| `f(x)` | `(f, x)` | one argument |
| `f(x y)` | `(f, x, y)` | positional arguments |
| `f([x y])` | `(f, [x, y])` | a list argument |
| `f({a: 1 b: 2})` | `(f, {a: 1, b: 2})` | a record argument — named arguments |
| `f()` | `(f)` | zero arguments |

```console
$ echo 'f(x y)' | wavelet read
(f, x, y)
$ echo 'f()' | wavelet read
(f)
```

As data, a call is just a tuple: `log("hi")` is the WAVE tuple `(log, "hi")`. In evaluation position a parenthesized form is always a call; the same tuple is plain data under `Quote`. A standalone `(a)` is therefore a one-element tuple — a zero-argument call of `a` — not a transparent grouping.

## 3. Qualified calls

A `/` between two names is a qualified reference: the name after the slash, taken from the import aliased by the name before it. WIT identifiers contain no slash, so this is a Wavelet-only token, and it heads a call exactly like an ordinary identifier:

| You write | Canonical WAVE form | |
|---|---|---|
| `kv/get({...})` | `(kv/get, {...})` | name `get` from the import aliased `kv` |

```console
$ echo 'kv/get({bucket: b})' | wavelet read
(kv/get, {bucket: b})
```

## 4. Call chaining

A `.` immediately following a form, then a name and an attached `(`, is **call chaining**: `recv.name(args)` reads as the call `(name, recv, …args)` — the receiver becomes the call's *first* argument. Chains fold left-to-right, so each `.name(...)` wraps everything to its left. Like the attachment rule, this is whitespace-sensitive: the `.`, the name, and the `(` must each abut the token before them.

| You write | Canonical WAVE form | |
|---|---|---|
| `1.increment()` | `(increment, 1)` | receiver as the sole argument |
| `x.foo(a b)` | `(foo, x, a, b)` | receiver, then the explicit arguments |
| `foo(1 2 3).bar(4 5 6)` | `(bar, (foo, 1, 2, 3), 4, 5, 6)` | the call's result chains onward |

```console
$ echo '1.increment()' | wavelet read
(increment, 1)
$ echo 'foo(1 2 3).bar(4 5 6).baz(7 8 9)' | wavelet read
(baz, (bar, (foo, 1, 2, 3), 4, 5, 6), 7, 8, 9)
```

This is **pure rewriting**, not method dispatch: there are no methods and nothing is attached to the receiver. `1.increment()` and `increment(1)` are the same call. It just lets a pipeline read left-to-right instead of inside-out — `"hello".upper().str-cat(", world!")` is exactly `str-cat(upper("hello") ", world!")`.

## 5. TitleCase macro sugar

A **TitleCase** identifier — its first hyphen-separated word starts with a capital and contains at least one lower-case letter (`If`, `DefMacro`, `Try-let`) — is reader sugar for a macro call. This can't collide with ordinary identifiers: a WIT label word is all-lower or all-UPPER, so a Title-case *leading* word is syntactically free (`parse-JSON` and `HTTP-get` stay ordinary names). The token is **lower-cased whole** — interior capitals are *not* split into hyphens, and existing hyphens are kept — then gains a `-MACRO` suffix: `If` ↦ `if-MACRO`, `DefMacro` ↦ `defmacro-MACRO`, `Try-let` ↦ `try-let-MACRO`. A multi-word macro may be spelled either as one word or hyphenated, and is invoked exactly as it was defined, lower-cased: `DefMacro try-let …` ↦ `Try-let …`, `DefMacro trylet …` ↦ `TryLet …`.

A TitleCase head does **not** require parentheses. The reader looks up the macro's declared **arity** and consumes exactly that many following forms, folding the head and those forms into a tuple:

```wavelet
If eq([foo bar]) log("match") log("nope")
```

```console
$ echo 'If eq([foo bar]) log("match") log("nope")' | wavelet read
(if-MACRO, (eq, [foo, bar]), (log, "match"), (log, "nope"))
```

Three consequences:

1. **Define before use.** A macro must be in scope before it's read — a core form, defined earlier in the file, or imported. The reader processes a file top to bottom and always knows every visible macro's arity.
2. **Explicit parentheses override arity reading.** `Unquote(x)` and `If(c t e)` attach their arguments directly instead of scanning ahead. The lowered call `if-MACRO(c t e)` — canonical `(if-MACRO, c, t, e)` — is always available too, and that tuple is what macros emit.
3. **Nesting needs no delimiters, in any position.** A TitleCase head consumes its own arity recursively wherever it appears, not just as the final argument. Both `If Quote c t e` (nested in the first slot) and `Def run Fn {} If c a b` (nested in the last) read exactly as intended.

Variadic macros take a single list argument — arity stays fixed, the list flexes (`Do([a b c])`).

## At a glance

Every reader-sugar form, in one table:

| # | You write | Canonical WAVE form | |
|---|---|---|---|
| 1 | `[1 2 3]` | `[1, 2, 3]` | commas are whitespace |
| 2 | `f(x y)` | `(f, x, y)` | attachment rule — calls are tuples |
| 3 | `kv/get({...})` | `(kv/get, {...})` | qualified call |
| 4 | `x.f(y)` | `(f, x, y)` | call chaining — receiver first |
| 5 | `If c t e` | `(if-MACRO, c, t, e)` | TitleCase macro |
