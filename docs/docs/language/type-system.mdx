# The type system

Wavelet is **statically typed**. Every expression has exactly one
[WIT](https://component-model.bytecodealliance.org/design/wit.html) type, decided
before any code runs. The compiler runs a checker pass over your whole program on
every path — `wavelet run`, `wavelet build`, `wavelet wit`, and the playground —
and an ill-typed definition is a **compile error even if nothing calls it**.
There is no runtime type error to discover at the wasm boundary — if it builds,
it is well-typed.

The system is **monomorphic**: there are no type variables in the language you
write or in the WIT it produces. Generics live entirely at compile time
(`Derive`, functors), and what ships is always concrete.

## Two rules

Checking is **bidirectional**, which is just two rules working together:

1. **Synthesis** — some expressions tell you their own type. An atom
   (`true`, `'x'`, `"hi"`), a bound name, or a call all *produce* a type from the
   bottom up.
2. **Checking** — some positions *expect* a type, and push it inward. A function
   parameter's declared type, a `Let` binding, the branches of an `If`, and the
   clause results of a `Match` all flow an expected type down into the
   sub-expression.

Two consequences fall straight out of "every expression has exactly one type":

- The branches of an `If` must share a type — `If b 1 "s"` is rejected, because
  an `s64` and a `string` have no common WIT type.
- The result clauses of a `Match` must likewise all unify to one type.

Because checking is **total**, names are resolved statically too: an unbound name
or a wrong-typed use inside a never-called function is still a compile error.

## Numeric literals and `The`

A bare numeric literal has no fixed type on its own — it takes its type from
context and is **range-checked** against it. `100` is a fine `u8`; `300` is not:

<Playground id="sf-the-bad" />

When there is no context to resolve a literal, it **defaults**: integer literals
default to `s64`, fractional literals to `f64`.

The `The` form ascribes a type to an expression. Use it to pin a literal's type,
to drive a range check, or — as you'll see below — to choose between overloads by
the result you want:

<Playground id="sf-the-ok" />

A literal that cannot fit the ascribed type is a compile error (`The u8 300`,
`The u8 -1`, `The s32 1.5` are all rejected).

## Overloading

Several `Def`s sharing a name form an **overload set**. Each call site is
resolved statically, first by the WIT types of its arguments:

<Playground id="ty-overload-int" />

Resolution doesn't depend on definition order — the other member is just as
reachable:

<Playground id="ty-overload-str" />

When the arguments don't decide (for instance, two zero-argument members), the
**expected return type** breaks the tie. Supply it with `The`:

<Playground id="ty-the-return" />

With neither arguments nor an expected type to disambiguate, the call is
**ambiguous** and is a compile error — you fix it by adding a `The`.

WIT has no overloading, so an *exported* overload set is **name-mangled at the
component boundary**: each concrete member gets a distinct export. An `eq` over a
`point` type, for example, is exported as `eq-point`.

A call that has no matching overload at all is simply ill-typed, even when it sits
in an uncalled function:

## Derive

`Derive` generates the standard operations for a type so you don't write the
boilerplate by hand. It takes a set of derivers and a type, and expands — at
compile time — into ordinary monomorphic definitions:

```wavelet
DefType point {x: s32 y: s32}
Derive {Eq Ord Show} point
```

This produces concrete operations specialized to `point` (an `eq` and a `show`
for `point`, an ordering comparison, and so on). The available derivers are
`Eq`, `Ord`, `Show`, and `Hash`. Because the generated definitions are just
ordinary overload-set members, exporting them mangles their names exactly like
any other export (`eq-point`, `show-point`, …).

`Derive` is a file-level macro: it runs during compilation of a component, not in
the interpreter playground.

## Functors

A **functor** specializes a generic data structure to a concrete element type at
compile time. You instantiate one with a parameterized `Import`, naming the
package, the element type, and a local alias:

```wavelet
Import {pkg: "wavelet:coll/set" elem: point as: pts}
```

This synthesizes a **concrete, per-element WIT interface** — instantiating `set`
at `point` yields a `point-set` interface. Nothing generic survives into the
emitted world: a second instantiation at, say, `string` would produce a separate
`string-set` interface. WIT cannot write `resource set<t>`, so the functor writes
the monomorphic interface for you, once per element type you ask for.

The instantiation also brings the functor's operations into scope, qualified by
the alias. For the `set` functor these are `pts/new`, `pts/add`, `pts/contains`,
and `pts/size`, with exactly the signatures the synthesized resource promises:

```wavelet
Let {s: pts/new()}              // new() -> set        (an empty set)
  Do [ pts/add(s p)             // add(set, elem)       (returns unit)
       pts/contains(s p)        // contains(set, elem) -> bool
       pts/size(s) ]            // size(set) -> u32
```

`new()` hands back a fresh, mutable set; `add` mutates it in place and is a no-op
for an element already present (membership uses the same equality `Derive {Eq}`
gives the element type); `contains` and `size` observe it. These run under
`wavelet run` and the playground today.

:::note Building functors

`wavelet wit`, `wavelet run`, and now `wavelet build` all support functors. The
backend emits the synthesized per-element interface as a **real exported WIT
`set` resource**, at interpreter parity: any element type (primitive, string,
record, and compound list/tuple) works, and you can instantiate the functor
multiple times in one world. Membership uses the same structural equality the
interpreter does, so a built component dedups exactly like `wavelet run`.

The single limitation is shaping, not semantics: an export that *returns the
`set` handle* (`own<set>`) over a **local-record** element would make the
element's `api` interface and the resource's interface each `use` the other — a
WIT interface cycle, which WIT cannot express, so it is rejected with a clear
error. The buildable shapes are therefore (1) exports that derive an ordinary
result from the set (e.g. `-> u32`), and (2) handle-returning exports whose
element is a primitive or string.

:::

## Worked example: source to WIT

Putting it together — a derived record type and a `set` of it — and the WIT the
compiler synthesizes. The headline is that **nothing in the WIT is generic**:

```wavelet
Package "demo:geo@0.1.0"

DefType point {x: s32 y: s32}
Derive {Eq Ord Show} point

Import {pkg: "wavelet:coll/set" elem: point as: pts}

Export nearest-set
Def nearest-set Fn {ps: list(point)}
  Let {s: pts/new()}
    Do [ each(ps Fn {p} pts/add(s p))
         s ]
```

synthesizes a world whose every signature is a concrete, ordinary WIT function or
type:

```wit
package demo:geo@0.1.0;

interface api {
  record point { x: s32, y: s32 }
  eq-point: func(a: point, b: point) -> bool;
  compare-point: func(a: point, b: point) -> s32;
  show-point: func(v: point) -> string;
  nearest-set: func(ps: list<point>) -> point-set.set;
}

// the set functor, instantiated at `point`
interface point-set {
  use api.{point};
  resource set {
    constructor();
    add: func(value: point);
    contains: func(value: point) -> bool;
    size: func() -> u32;
  }
}

world geo {
  export api;
  export point-set;
}
```

The record, the derived operations, and the export all land in a single `api`
interface, while each functor instantiation gets its own interface (which `use`s
the element type from `api`). The overload set `eq` lowers to the distinctly
named `eq-point`, the derived operations become concrete functions, and the `set`
functor becomes the concrete `point-set` interface. A generic set would have
wanted `resource set<t>` with `add: func(value: t)`, which WIT simply cannot
express — the functor produces a concrete interface instead.

One caveat on this particular example: `nearest-set` *returns* the `set` handle
over the local record `point`, which is exactly the shape `wavelet build` rejects
(the `api` and `point-set` interfaces would each `use` the other — a WIT
interface cycle). `wavelet wit` and `wavelet run` still synthesize and execute it,
so it illustrates the synthesis above; to *build* a `point` set, derive an
ordinary result from it (e.g. `-> u32`) rather than returning the handle.

(`each` here is illustrative iteration over the input list, applying `pts/add`
for its side effect; it is not yet a builtin. To run a set today, iterate with
`map`/`fold` or with explicit `pts/add` calls.)

## A program you can build

The example above *synthesizes* — `wavelet wit` prints it and `wavelet run`
executes it — but its `nearest-set` returns the `set` handle over a record, the
one shape `wavelet build` rejects. Here are the two shapes that **do** build,
each shown with the WIT the backend actually emits (run `wavelet build` then
`wasm-tools component wit` on the result to reproduce these verbatim).

### Derive an ordinary result

The simplest buildable functor program never hands the `set` back across the
boundary — it derives a plain value from it. Reusing the `point` set, an export
that counts distinct points returns a `u32`:

```wavelet
Package "demo:geo@0.1.0"

DefType point {x: s32 y: s32}
Derive {Eq Ord Show} point

Import {pkg: "wavelet:coll/set" elem: point as: pts}

Export count-distinct
Def count-distinct Fn {}
  Let {s: pts/new()}
    Do [ pts/add(s {x: 1 y: 2})
         pts/add(s {x: 3 y: 4})
         pts/add(s {x: 1 y: 2})   // a structural duplicate of the first
         pts/size(s) ]
```

`wavelet build` emits a component whose `api` exports `count-distinct` as an
ordinary `u32` function; the `set` resource stays entirely inside the guest:

```wit
package demo:geo@0.1.0;

interface api {
  record point { x: s32, y: s32 }
  eq-point: func(a: point, b: point) -> bool;
  compare-point: func(a: point, b: point) -> s32;
  show-point: func(v: point) -> string;
  count-distinct: func() -> u32;
}

interface point-set {
  use api.{point};
  resource set {
    constructor();
    add: func(value: point);
    contains: func(value: point) -> bool;
    size: func() -> u32;
  }
}

world geo {
  export api;
  export point-set;
}
```

Calling `count-distinct` returns `2`: the two `{x: 1 y: 2}` points are
structurally equal, so the set dedups them — exactly the answer `wavelet run`
gives, because membership uses the same structural equality on both paths.

### Return the handle (primitive or string element)

When the element is a primitive or a string — not a local record — there is no
interface cycle, so an export *can* hand the `set` itself back as an `own<set>`
handle for a host to keep and query:

```wavelet
Package "demo:app@0.1.0"

Import {pkg: "wavelet:coll/set" elem: s32 as: ints}

Export build-ints
Def build-ints Fn {}
  Let {s: ints/new()}
    Do [ ints/add(s 1)
         ints/add(s 2)
         ints/add(s 1)            // duplicate; dedups away
         s ]                      // return the set itself
```

The emitted `api` returns the handle. Each functor interface names its resource
`set`, so the `use` is aliased (`set as s32-set-handle`) — which is also what lets
a single world instantiate the functor at several element types without the names
colliding:

```wit
package demo:app@0.1.0;

interface s32-set {
  resource set {
    constructor();
    add: func(value: s32);
    contains: func(value: s32) -> bool;
    size: func() -> u32;
  }
}

interface api {
  use s32-set.{set as s32-set-handle};
  build-ints: func() -> s32-set-handle;
}
```

A host calls `build-ints`, then invokes `size`/`contains` on the returned
resource (here `size` is `2`; `contains 1` and `contains 2` are true, `contains 9`
is false) and finally drops the handle to run the resource's destructor. This is
exactly what `tests/backend_functor.rs` does in-process against the wasmtime host
to prove the built component matches the interpreter — over `s32`, `string`,
`point` records, and `list` elements, and with two instantiations in one world.
