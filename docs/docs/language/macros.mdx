# Macros

A macro is a **compile-time function from argument forms to a result form**. Because Wavelet is homoiconic, "a form" is an ordinary value, so a macro is just a function over values that happens to run during reading/expansion.

## `DefMacro`

`DefMacro name {params} body` binds the macro's parameters to the **unevaluated** argument forms, runs the body, and replaces the call with the form the body returns.

<Playground id="macro-swap" />

Once defined, the macro's **arity** is known to the reader, so its TitleCase form reads without parentheses (here `Swap` consumes two forms). Macros must be [defined before use](./syntax#the-macro-sugar).

## Quasiquote, unquote, splice

The template trio builds result forms ergonomically:

- [`Quasi`](./special-forms#quasi) reproduces a form as data.
- [`Unquote`](./special-forms#unquote) evaluates a hole and inserts the result.
- [`Splice`](./special-forms#splice) evaluates a hole to a list and splices its elements into the surrounding sequence.

## Hygiene and `gensym`

Expansion is **unhygienic**, in the Common Lisp tradition: names a macro introduces can capture or be captured. The discipline is [`gensym`](../stdlib#meta--reflection), which mints a fresh, collision-free name each call.

<Playground id="macro-gensym-three" />

Each call returns a distinct name, so a macro can introduce identifiers that are guaranteed not to clash with the user's:

<Playground id="macro-gensym" />

:::caution[Binding keys must be literal names]
The reader requires record keys â€” and therefore [`Let`](./special-forms#let) binding names and [`Fn`](./special-forms#fn) parameter names â€” to be **literal identifiers** in source. You cannot `Unquote` a `gensym`'d name into a binding *position* (e.g. `Let {Unquote(g): â€¦}`); `gensym` is for fresh names used in *reference* and *case* positions. A reader that accepts computed binding keys, and a full hygiene layer, are **not yet implemented** â€” see the [roadmap](../roadmap). For now `gensym` is the discipline against accidental capture.
:::

## Error propagation without early return

The core has no early return, so error propagation is a *binding* macro: it wraps the rest of the computation rather than escaping it. This is the `Try-let` pattern â€” Rust's `?`, reconstructed in user space from `Match` and quasiquote. Note the macro is *defined* under its lower-case name `try-let` and *invoked* through the TitleCase sugar `Try-let` (just as the core `If` invokes `if-MACRO`):

```wavelet
DefMacro try-let {binding body}        // binding is a one-field record form
  Let {name: rec-key(binding) expr: rec-val(binding)}
    Quasi Match Unquote(expr) [
      (ok(Unquote(name))  Unquote(body))
      (err(e)             err(e))
    ]

Def load-config Fn {path: string}
  Try-let {text: read-file({path: path})}
  Try-let {form: read(text)}
  ok(form)
```

You can build it live with [`rec-key`/`rec-val`](../stdlib#meta--reflection):

<Playground id="macro-trylet" />

:::caution[Convenience macros aren't built in]
`And`, `Or`, and `TryLet` are **illustrations, not standard-library functions** â€” Wavelet's standard library ships [builtins](../stdlib) but no pre-defined macros beyond the sixteen core forms. Define the ones you want with `DefMacro` (as above), or ship them as a [macro component](#macros-as-components) and import them.
:::

## Macros as components

Macros are **functions over a WIT type** (`tree â†’ tree`), so they need not be written in Wavelet, nor live in the same file. A component exporting `wavelet:meta/macros` is a **macro library**: Wavelet instantiates it *at compile time*, asks it which macros it provides, and routes expansion through it. Because it runs inside a wasm component, an untrusted macro is **sandboxed by construction** â€” it gets no capabilities the build doesn't hand it. This is the basis of Wavelet's [supply-chain story](../supply-chain).

### Importing a macro library

`Import {pkg: "â€¦" macros: true}` loads a macro library and makes its macros available, exactly like local `DefMacro`s â€” the imported names read paren-free at their published arity:

```wavelet
Import {pkg: "acme:html/dsl" macros: true}   // load this component's macros

Export {name: page params: {title: string} result: string}
Def page Fn {title: string}
  Element "h1" title                          // 'Element' is a foreign macro
```

A `macros: true` import is **compile-time-only**: the library is instantiated and run during the build, then discarded. It is *not* a runtime dependency of your component and contributes nothing to the synthesized world â€” your shipped component never imports `wavelet:meta/macros`.

Wavelet resolves the library's `.wasm` two ways. Give an explicit path with `from:`, or drop the built component at the conventional location `wit/macros/<ns>-<name>.wasm` and let resolution find it:

```wavelet
Import {pkg: "acme:html/dsl" macros: true from: "vendor/dsl.wasm"}
Import {pkg: "acme:html/dsl" macros: true}    // resolves wit/macros/acme-html.wasm
```

### The `wavelet:meta/macros` interface

A macro library is any component that exports `wavelet:meta/macros@0.1.0`:

```wit
interface macros {
  use code.{tree};
  manifest: func() -> list<tuple<string, u32>>;          // (name, arity) pairs
  expand: func(name: string, args: tree) -> result<tree, string>;
}
```

`manifest()` tells the reader each macro's name and arity (so its TitleCase form reads at the right arity, top-to-bottom, like a local `DefMacro`). `expand(name, args)` does the work: Wavelet marshals the call's argument forms into a `tree` (a flat arena encoding of a form â€” `node`s and a root index, the wire form of homoiconic code), calls `expand`, and lifts the returned `tree` back into the form tree, recursing to a fixpoint. A macro that can't expand a call returns an `err` string, which surfaces as a build error.

Because the contract is just this WIT interface, you can write a macro library in **any language** that compiles to a component â€” say a `Re` macro that builds a regex DFA at build time in Rust â€” and it behaves exactly like a native one.

### Writing a macro library in Wavelet

The simplest way to ship a macro library is to write it in Wavelet. A file whose top level is a `Package` plus only `DefMacro`s (no `Export`/`Def`s) **is** a macro library â€” `wavelet build` compiles it into a `wavelet:meta/macros` component:

```wavelet
Package "acme:html@0.1.0"

DefMacro element {tag body}
  Quasi html-node Unquote(tag) Unquote(body)

DefMacro unless {cond body}
  Quasi If Unquote(cond) {} Unquote(body)
```

```console
$ wavelet build src/html.wvl -o out      # emits out/acme-html.wasm
```

Drop `out/acme-html.wasm` at `wit/macros/acme-html.wasm` (or point `from:` at it) and another file can `Import {pkg: "acme:html/dsl" macros: true}` and use `Element`/`Unless` as if they were local. The library bundles the Wavelet interpreter and runs your `DefMacro`s inside the sandbox at the consumer's build time.

### Qualifying and aliasing

Imported macros are available **bare** (`Element`) and **qualified** (`dsl/Element`, using the import's alias). When a bare TitleCase name is provided by two imports â€” or by an import *and* a local `DefMacro` â€” using it bare is an error that tells you to qualify or alias; the qualified spelling always resolves:

```wavelet
Import {pkg: "acme:html/dsl" as: dsl macros: true}
Import {pkg: "acme:svg/dsl"  as: svg macros: true}

dsl/Element "h1" title       // unambiguous â€” picks the acme:html macro
svg/Element "circle" attrs   // unambiguous â€” picks the acme:svg macro
```

The ambiguity is only reported when the colliding bare name is actually **used**; a qualified use is never ambiguous.
