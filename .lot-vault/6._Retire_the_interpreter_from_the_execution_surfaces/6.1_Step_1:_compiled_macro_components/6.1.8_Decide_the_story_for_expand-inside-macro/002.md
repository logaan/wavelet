**Question: when a compiled macro's body calls the `expand` builtin
(builtins.rs:365), what happens?**

Options:

1. **Reject with a clear diagnostic (recommended).** Compiling a macro whose
   body calls `expand` fails with "expand is not available inside compiled
   macros", and the limitation is documented. Macro components stay pure
   tree-to-tree functions with no host entanglement. It's rare — the shipped
   examples don't use it.
2. **Route it back to the host expander.** The macro component imports an
   expand function the host provides. Preserves the capability, but re-enters
   the host expander mid-expansion (reentrancy through shared expander state)
   and pins every macro component's ABI to an unstable host-internal import.
3. **Compile the expander into each macro component.** Self-contained but
   heavy, and it freezes a copy of the expander's semantics into every macro
   at its build time, so macros can silently disagree with the compiler that
   loads them.

Recommendation: **option 1 now**, documenting option 2 as the future path if a
real macro ever needs staged expansion. The revisit trigger is a concrete use
case, not speculation — and a use case would also tell us which expander state
the callback actually needs to see, which is the hard design question option 2
currently hand-waves.
