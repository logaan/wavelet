# Interview: scoping the WIT round-trip suite

Recon first, so these questions are grounded:

- Toolchain is ready: cargo-component 0.21, wasmtime 44, wasm-tools 1.235, wac 0.10, wkg 0.15, wit-bindgen 0.43.
- The referenced WIT page covers: all primitives, list, option, result (all 4 forms), tuple, record, variant, enum, flags, resources (constructor/methods/statics/borrow), type aliases, functions, interfaces, use, worlds, packages. It does NOT cover streams/futures/async or fixed-length lists, so I'm treating those as out of scope.
- Wavelet's known boundary gaps (dev-notes/gaps.typ) that this suite will definitely hit: constructing user variant/enum cases in compiled code, variant/enum > 256 cases and flags > 32 members, user-declared resource types.

## Questions

**Q1. Are resources in scope?** Resources are handles, not values, so "transform and
return" doesn't apply directly — they need their own shape (e.g. a `counter`
resource with a seeded constructor, a method that advances it, a static fn, and
functions passing owned + borrowed handles in both directions). Wavelet has no
general user-declared resources yet, so including them guarantees a large step-4
work item.
*Recommendation:* include them, but in a separate interface
(`roundtrip:suite/resources`) from the plain data-type interface
(`roundtrip:suite/values`), so an implementation can pass the values suite
independently.

**Q2. Where does the Rust component live?**
*Recommendation:* a self-contained crate in this repo, e.g. `conformance/` —
its own Cargo project with no wavelet dependency and its own README, easy to
extract to a separate repo later. Alternative: its own repo from day one.

**Q3. PR staging.** Natural stages: (1) Rust component + WIT world + self-test
harness; (2) Wavelet counterpart + both-directions test matrix + shortcomings
list posted here; (3+) one PR per shortcoming fixed.
*Recommendation:* one PR per stage as above, rather than a single mega-PR.

**Q4. After the shortcomings list (step 3 → 4).** Some shortcomings map to known
big features (user variant/enum construction at the boundary; resources).
*Recommendation:* I post the list here, keep implementing the tractable ones
autonomously, and stop before any that need a language-design decision from you.

## Proposed defaults (veto anything, otherwise these stand)

- **World shape:** package `roundtrip:suite`. Interface `values` with one
  function per feature (`u8-round-trip: func(v: u8) -> u8`, …) covering: every
  primitive; list; option (some and none across the seed sets); all four result
  forms; tuples; record; variant; enum; flags; a type alias; nested compounds
  (list<record>, option<variant>, result<tuple, enum>, list<list<u8>>);
  multi-param and zero-param/zero-result functions. The test component both
  imports and exports `values` (that's the symmetry) and exports a `runner`
  interface: `run: func() -> result<_, list<string>>` returning one message per
  failed check.
- **Transform convention** (documented in the WIT doc comments so any
  implementation can follow it): ints +1 wrapping; floats +1.0; bool negated;
  char next Unicode scalar; string appends "!"; compounds recurse elementwise;
  enums rotate to the next case; variants keep the case and transform the
  payload; flags toggle every member; bare `result` flips ok↔err, payloaded
  results keep the side and transform the payload.
- **Seeds:** two seed tables in the crate selected by a cargo feature; the build
  script emits `roundtrip-a.wasm` and `roundtrip-b.wasm`.
- **Harness:** compose caller+callee with `wac plug`, execute with
  `wasmtime run --invoke`; a script drives the full matrix — {rust-a, rust-b,
  wavelet} as caller × as callee (seeds only matter on the caller side).

A one-line "all recommendations" reply unblocks stage 1 immediately.
