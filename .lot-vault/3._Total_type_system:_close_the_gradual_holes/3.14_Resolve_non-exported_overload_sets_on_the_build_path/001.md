# 3.14 Resolve non-exported overload sets on the build path

Found by the F4 execution differential harness (1.1, PR #36,
tests/differential.rs).

## The gap

`build_files` runs the *pure* checker (`check_program`) but never
`resolve_overloads` — deliberately, because exported overload sets are
emitted by keying on the intact sets (`info.fn_defs`/`overload_bodies`),
which the rewrite would destroy. But a **non-exported** overload set gets
neither treatment: `wit::collect` records only one `(params, body)` per name
in `FileInfo::defs` (last definition wins), so *every* call site dispatches
to the last-defined member regardless of the argument's static type or a
`The`-expected type.

The interpreter path (eval_snippet / run) does run `resolve_overloads`,
which rewrites each call to the correct uniquely-named member — so the two
execution surfaces disagree.

## Observed divergences (docs examples, via the harness)

- `ty-overload-int`: `show(5)` — interpreter "int", compiled "str"
- `ty-overload-str`: `show("hi")` — interpreter "str", compiled "int"
- `ty-the-return`: `The s64 make()` — interpreter 1, compiled "x"

## Fix sketch

Either (a) run the overload-resolution rewrite on the build path for calls
while keeping the original sets intact for `wit::collect`'s export keying
(rewrite call *sites* only, or rewrite after collect), or (b) teach emit's
internal-call dispatch to select the member by the checker's resolved type,
the way exported sets already mangle per member.

## Exit gate

Remove the three OVERLOAD entries from SKIP in tests/differential.rs; the
harness then locks the behaviour.
