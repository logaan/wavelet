**Question: what replaces the uniform-box bump allocator for the typed heap —
arenas, refcounting, or wasm-GC — and, equivalently, where (if anywhere) may
headers still exist?**

Options:

1. **Arena-per-call, headerless everywhere (recommended, contingent on 2.3
   going to defunctionalization).** Static types drive per-type layout; no
   object carries a header; every allocation bump-allocates into an arena that
   is reset when the export call returns (component instances are short-lived,
   so per-call reset bounds everything). Simplest to build, fastest in the
   common case, and canonical layout holds unconditionally.
2. **Refcounting with per-type layout maps.** Counts have to live somewhere —
   headers (breaking canonical layout), fat pointers, or side tables — plus
   generated drop glue per type and a cycle story for recursive types. The
   payoff profile (long-lived instances with high churn) isn't the current
   workload.
3. **wasm-GC structs for the non-ABI residue, linear memory for ABI data.**
   Only necessary if 2.3 keeps a private closure representation — closures
   would be the sole headered family. Adds an engine/tooling dependency on
   wasm-GC for one value kind.
4. **Full wasm-GC heap.** Everything lives in GC structs and must be copied
   into linear memory at every canonical-ABI boundary; worst of both worlds.

Recommendation: **option 1**, with two riders. (a) It is contingent on 2.3:
defunctionalized closures are plain variants, leaving zero non-ABI residue and
making "no headers anywhere" achievable — if 2.3 instead keeps a private
representation, take option 3 for closures only. (b) The known failure mode —
a single export call that allocates unboundedly in a loop — is accepted as out
of scope for now; the escape hatch is region-reset points inside a call, which
headerless layout deliberately leaves open and which can be added later without
changing any layout decision made now.
