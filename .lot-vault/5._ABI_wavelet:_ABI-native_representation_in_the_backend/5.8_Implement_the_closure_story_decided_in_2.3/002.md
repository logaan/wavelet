2.3 is decided (Logan, 2026-07-03): defunctionalization, with expand-time
combinators as the first cut and resource-lifting as a non-gating extension
(now child 5.8.1). This Thing's scope is therefore concretely:

1. Per arrow type used in the component: generate the variant (one case per
   `Fn` of that type; payload = concretely-typed captures) and the dispatch
   function; apply-sites call dispatch. A single-case variant devirtualizes
   to a direct call.
2. A **deterministic naming scheme** for the generated variants and dispatch
   functions, so synthesized WIT is stable across edits (design risk flagged
   in update 003 on 2.3 — settle it in this Thing's review).
3. **Reject function types in cross-component signatures with an explicit
   diagnostic.** Today an arrow type in an Export/Import signature is
   unspellable by accident; make it a deliberate, well-worded "not yet"
   (pointing at expand-time combinators as the supported path). Lifted when
   5.8.1 lands.
4. **Oracle parity**: interp.rs currently passes closures across the runner's
   simulated component boundaries freely (imports resolve by package id in
   one heap). The interpreter/check.rs must model the same boundary rule as
   the backend (reject escaping function values) before or with the emit
   change, or the oracle and the backend disagree on exactly the programs
   this decision governs.

Replaces the old either-way framing in this Thing's note: the "if rejected:
fence off a private representation" branch is dead, and 5.1 may assume zero
non-ABI residue (headerless layout unblocked, pending Logan's 5.1 decision).
Absorbed parity gap "Support qualified symbols as bare values" is the
degenerate case: a bare name is a zero-capture case of its arrow type's
variant.
