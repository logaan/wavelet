# 5.8.1 Resource-lifting for escaping closures (extension, not gating 5.8)

The designed extension from the 2.3 decision (accepted 2026-07-03): when a
function type must genuinely cross a component boundary (callbacks, handlers,
subscriptions — not combinators, which are expand-time per 5.7.1), the
compiler lifts the escaping closure into a generated single-method resource:

- Per escaping arrow type, emit `resource fn-<sig> { call: func(...) -> ...; }`
  in the synthesized WIT; the exporting component implements `call` as the
  generated dispatch function (from 5.8) applied to the variant stored in the
  resource entry. The internal representation never leaks.
- Inbound, a lone-`call` resource ducks in as a function value: the arrow
  type's dispatch variant gains one case holding the foreign handle (a plain
  ABI value — headerless layout is preserved).
- **Borrow-only first**: `borrow<>` scopes the handle to the call, which is
  the callback pattern and needs no lifetime story. `own` handles (callee
  retains the closure) require the general resource drop semantics — rides
  on 4.5, land them together.
- Lifts 5.8's explicit "function types rejected in cross-component
  signatures" diagnostic when it lands.

Precedent in-tree: the set functor already emits a compiler-generated,
component-implemented resource from a shared op-table (wit.rs) — the same
shape this needs.

Known limits (recorded in the 2.3 decision): handles are per-instance
identity, not serializable values, so this does NOT serve 8.2; and each
callback is a canonical-ABI boundary call with copies — there is currently
no scheduled fusion/inlining mitigation, which is why combinators stay
expand-time.

Trigger: the first real cross-component callback need. Do not start this
speculatively.
