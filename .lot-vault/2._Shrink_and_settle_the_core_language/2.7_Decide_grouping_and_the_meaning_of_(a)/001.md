# 2.7 Decide grouping and the meaning of (a)

Broken out of 2.4 (item 4, grouping semantics), per Logan's update 003 there.

## Where the language stands today

The reader gives `( ... )` exactly one meaning: build a tuple. `()` reads as
`Tup[]`, `(a)` as `Tup[a]`, `(a b)` as `Tup[a, b]` (reader.rs,
`parse_parens`). In evaluation position a tuple is a call — head applied to
the rest — so a free-standing `(a)` already IS a zero-argument call of `a`,
exactly the normal lisp reading. Under `Quote` the same form is plain data: a
one-element tuple. This is documented behaviour
(docs/docs/language/syntax.mdx: "`(a)` — a zero-argument call of `a` (no
transparent grouping)"; reader-sugar.mdx says the same).

So there is no grouping form in the language today, and nothing needs one:
Wavelet has no infix operators or precedence levels, which is the only job
grouping parens do in other syntaxes. The question 2.4's options update
raised was whether to *change* this to transparent grouping, where `(a)` ≡ `a`.

## Question

Should free-standing `(a)` stay what it is — a one-element tuple, i.e. a
zero-argument call in evaluation position — or become transparent grouping
(`(a)` ≡ `a`)?

## Options

1. **Keep the current semantics: `(a)` is `Tup[a]`, a zero-arg call
   (recommended).** One rule with no context-sensitivity: parens always build
   a tuple, whatever the length, wherever they appear; being in evaluation
   position is what makes a tuple a call. Quoted and macro-manipulated trees
   see exactly the structure the programmer wrote. `(a x y)` calls `a` with
   two arguments, `(a)` calls it with none. No implementation or docs work:
   this is already the shipped, documented behaviour.

2. **Transparent grouping: the reader erases the parens, so `(a)` ≡ `a`.**
   What 2.4's options update recommended. In a syntax with no precedence
   there is nothing for grouping to disambiguate, so the form's only
   observable effect is erasure: `Quote (a)` and `Quote a` would read
   identically, one-element tuples would become unspellable as data, and a
   zero-argument call of a computed (non-named) head would need a new
   spelling. It resolves the "context-sensitivity" worry by deleting
   information the programmer wrote rather than by making the rules uniform
   — and the rules are already uniform.

3. **Make free-standing `(a)` a read error, requiring the attached `a()`
   spelling for zero-arg calls.** Would catch stray parens, but adds a
   length-1 special case to an otherwise uniform reader rule and makes
   one-element tuple data unspellable — the wrong trade in a homoiconic
   language.

## Recommendation

**Option 1: keep `(a)` as a zero-argument call and close this with no
language change.** The worry recorded in 2.4 — that the call reading makes
`(a)` "context-sensitive" in quoted trees — has it backwards. Under the
current rule `(a)` denotes the same tree everywhere (`Tup[a]`); only
evaluation gives any tree meaning, and that is true of every form in the
language. It is the grouping reading that would introduce a read-time
landmine for quoting and macros, by erasing structure before macros ever see
it. Grouping earns its keep only in syntaxes with precedence to override;
Wavelet has none.

If accepted, the follow-through is nil: nothing changes in the reader, docs,
or grammars. (`()` as `Tup[]` — currently an eval-time error — is untouched
by this decision; with 2.4's unit-value drop it stays an error rather than
becoming a unit literal.)
