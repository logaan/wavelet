Five independent keep/drop calls; each can be decided separately. Question,
options, and recommendation per item:

1. **The unit value — keep or drop?** WIT has no unit type: "returns nothing"
   is an empty result list, not a value. Options: (a) drop the user-facing unit
   value — a no-result function simply has no result, and effect-position
   expressions need no value to name; (b) keep it, lowering to an empty record
   at boundaries. **Recommend (a), drop.** It's a value with no ABI
   counterpart; anyone who genuinely needs a placeholder type can declare an
   empty record in userland.

2. **`///` doc comments — bless or drop?** The lexer has already demoted `///`
   to an ordinary comment. Options: (a) bless `///` as documentation attached
   to the following definition; (b) leave it a plain comment. **Recommend (a),
   bless.** It's the cheapest item on this list with an exact WIT counterpart —
   WIT itself uses `///` docs — so it can flow straight through WIT synthesis,
   and it feeds LSP hover. Timing: wire it up whenever `wit.rs` learns to emit
   docs; nothing blocks on it.

3. **`f(x y)` tuple-call sugar — keep or drop?** A second spelling for
   application, redundant with the plain form. Options: (a) drop it — one way
   to spell a call keeps the reader, printer, formatter, and all three
   highlighting grammars simpler; (b) keep for familiarity. **Recommend (a),
   drop.** Sugar that only duplicates an existing spelling is pure surface
   area; it can return later without breaking any program, whereas removing it
   later breaks code. (The `recv.name(...)` chained-call form is a separate
   question — judge it on whether the docs/examples actually lean on it.)

4. **Grouping `(a)` semantics — what does it mean?** Options: (a) pure
   grouping: `(a)` ≡ `a`, always, and a zero-argument call must be spelled
   explicitly; (b) parentheses around a lone form mean "call it". **Recommend
   (a), pure grouping.** The call reading makes `(a)` context-sensitive in
   quoted/macro-manipulated trees — a reader-level landmine in a homoiconic
   language. Explicit is cheap; ambiguity isn't.

5. **`Do`/`Match` as stdlib rather than special forms?** Options: (a) move
   both; (b) move `Do`, keep `Match` special; (c) keep both special.
   **Recommend (b).** `Do` is plausibly expressible as a bundled macro with no
   checker involvement. `Match` is not just sugar once 2.2.3 lands: under
   total types the checker needs it natively for exhaustiveness checking and
   case narrowing, so it is a core typing construct. (The parked 8.3 question
   about alternatives to the bundled stdlib doesn't block moving `Do` — it
   only affects where the macro lives.)

Sequencing note: all of these land naturally with the 3.9/3.10 strict-mode
burn-down; none needs its own project.
