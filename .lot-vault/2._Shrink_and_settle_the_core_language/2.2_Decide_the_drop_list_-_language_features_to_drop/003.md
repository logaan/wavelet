Each child now carries its decision as a question with options and a
recommendation — record each yes/no on the child itself. The recommended
ledger, one line each:

- **2.2.1 heterogeneous lists — drop.** Lists are `list<t>`; mixed shapes are
  `list<variant>`, mixed code-like data is `tree`.
- **2.2.2 spontaneous structural records — drop.** Nominal + inference; rows
  stay on the table only if the ceremony stings in practice.
- **2.2.3 tag-dispatch dynamism — drop all four behaviours.** Explicit
  conversions and known-shape scrutinees; literal-adoption sugar is a
  separable checker-side follow-up.
- **2.2.4 runtime The checks — drop.** `The` is pure static ascription;
  `safely` survives only at `read` and foreign-component boundaries.
- **2.2.5 runtime eval — drop.** `read` stays pure data; the dynamic island is
  parked pending a concrete use case; REPL unaffected.
- **2.2.6 gradual escape hatch — drop.** Inference failure is rejection; the
  gradual flag dies at the 3.9 flip.
- **2.2.7 first-class functions — moot on the recommended path.** 2.3 should
  go to defunctionalization; if it doesn't, fall back to top-level function
  references (no captures).

Pattern worth naming: every recommendation is the drop, because each feature
exists only to spend runtime tags, and 2.3's defunctionalization removes the
last reason ANY value needs a tag. Deciding these seven as a block is what
lets 5.1 take the headerless-arena story cleanly.
