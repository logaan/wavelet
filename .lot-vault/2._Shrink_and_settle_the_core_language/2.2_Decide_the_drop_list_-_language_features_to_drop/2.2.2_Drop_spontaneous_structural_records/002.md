**Question: must every record literal have a nominal type — declared via
`DefType` or inferable from its context — so ad-hoc structural bags of fields
go away?**

Options:

1. **Nominal + inference (recommended).** A record literal must check against a
   declared type, found either by ascription (`The`), by the position it's used
   in (parameter/result/field of a declared type), or by unambiguous inference.
   A bag of fields with no declared home is rejected.
2. **Row-typed records.** Row polymorphism keeps today's ergonomics (any literal
   is well-typed, functions can accept "records with at least these fields") at
   the cost of a genuinely bigger type system: row unification inside inference,
   harder error messages, and a nominalization step at every WIT boundary anyway
   (WIT records are named).
3. **Status quo untyped bags.** No canonical-layout representation; dies with
   the ABI goal regardless.

Recommendation: **option 1**, as the original note already leans. WIT records
are nominal, so nominal wavelet records erase to the ABI with zero adaptation —
rows would still have to be nominalized at every boundary. The asymmetry also
favours starting nominal: adding rows later is an extension, while shipping rows
first is hard to walk back. Revisit only if practice shows the declaration
ceremony genuinely stings.
