**Question: do all four runtime tag-dispatch behaviours become compile errors
or explicit conversions?** The four: mixed int/dec arithmetic (silent f64
promotion), `match` over values of statically unknown shape, bare symbols
doubling as untyped variant cases, and comparisons across types.

Options:

1. **Drop all four (recommended).** Arithmetic requires same-typed operands,
   with explicit `to-f64`-style conversions; a match scrutinee must have a
   statically known variant/record type; a bare symbol must resolve to a case
   of a declared variant known from context; cross-type comparison is a compile
   error.
2. **Drop the runtime dispatch, allow static sugar.** Same runtime semantics as
   option 1, plus checker-side conveniences where types are statically known —
   e.g. an int *literal* adopts the dec type in a dec context. No tags are read
   at runtime; this is purely a checker feature.
3. **Keep runtime dispatch.** Requires universal value tags; incompatible with
   canonical layout, so effectively not on the table.

Recommendation: **option 1 now**, treating option 2's literal-adoption as a
separable ergonomic follow-up to decide only when the strictness demonstrably
hurts. Landing 1 strictly first keeps the 3.9/3.10 burn-down mechanical: every
new failure has a rote fix (insert a conversion, declare a type), never a
semantics debate. This is the widest-touching drop, so mechanical matters.
