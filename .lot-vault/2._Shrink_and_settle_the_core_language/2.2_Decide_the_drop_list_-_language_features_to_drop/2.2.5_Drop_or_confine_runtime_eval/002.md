**Question: what survives of runtime eval — does `read` become purely a data
operation, with no way to evaluate a runtime-constructed tree?**

Options:

1. **Drop runtime eval entirely (recommended).** `read` produces trees; trees
   are data to walk, transform, and print — enough for data formats and for
   feeding the compile-time expander. All *evaluation* is compile time: macros
   cover metaprogramming, and the REPL is unaffected because it already
   recompiles a scratch component per definition.
2. **Confine eval to a dynamic-island component.** Preserves the capability
   behind a typed boundary, but means keeping an evaluator alive as a shippable
   runtime artifact — a second execution surface colliding head-on with goal 6
   (retire the interpreter), maintained for a use case we don't currently have.
3. **Keep eval in the language.** The result type of evaluating a runtime tree
   is statically unknowable; incompatible with total types.

Recommendation: **option 1.** No shipped example needs runtime eval and the
REPL story doesn't change. Park the dynamic island as an 8.x question and
reopen it only against a concrete use case — a real use case is also the only
thing that can answer the genuinely hard part of option 2, namely what the
island's boundary types would even be.
