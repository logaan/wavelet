**Question: do we drop heterogeneous lists, so every list is a genuine `list<t>`?**

Options:

1. **Drop (recommended).** A list expression must have a single element type;
   mixing is a compile error. Mixed *shapes* are spelled `list<some-variant>`;
   mixed *code-like* data lives in `tree`.
2. **Keep, via compiler-synthesized variants.** The checker infers an anonymous
   variant of the observed element types. Representable under canonical layout,
   but the inferred types are unstable (adding one element to a literal changes
   a type that may appear in WIT signatures and error messages), and it hides
   exactly the intent that a declared variant states.
3. **Keep, via boxed/tagged elements.** Requires uniform boxes — this is the
   representation goal 5 exists to remove, so it amounts to vetoing ABI wavelet
   for lists.

Recommendation: **option 1, drop.** The checker already degrades heterogeneous
lists to Unknown, and the only sanctioned producer of mixed lists — quoted data
— is moving to `tree`, which removes the remaining pressure. The cost is minor
ceremony: declaring the variant you actually mean. Options 2 and 3 each
preserve the surface at the price of the thing the plan is trying to buy
(stable, ABI-native types).
