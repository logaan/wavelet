**Question (contingent — live only if 2.3 rejects defunctionalization): what do
first-class functions shrink to?**

Options:

1. **References to top-level defs (recommended, if we land here).** A function
   value is a reference to a top-level definition — no captured environment, so
   it's representable as a plain discriminant/index. Covers the bulk of
   map/filter/callback usage; where a capture is wanted, the caller passes the
   data alongside explicitly.
2. **Drop function values entirely.** Every callback site hand-writes the
   variant-plus-dispatch pattern. Honest and zero magic, but pushes recurring
   boilerplate onto exactly the most common higher-order uses.

Recommendation: **decide 2.3 first, and decide it in favour of
defunctionalization — which makes this Thing moot.** That is the recommended
path (see the write-up on lot:033iWUQfu2XoRGFBRkLq5R). If defunctionalization
is rejected, take option 1: it is defunctionalization's degenerate no-capture
case and can be widened later without breaking programs.

Unchanged either way: passing an *overloaded* name as a value stays permanently
out — an overload set has no single type, so no single representation (already
deferred as Phase D).
