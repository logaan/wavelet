**Question: which surface names change to align with component-model
vocabulary — in particular, is `Package` renamed, and to what?**

Context for the call: today `Package "demo:app@0.1.0"` heads a file and the
compiler derives the component's WIT world from what follows. In the component
model's own glossary: a *package* is a versioned collection of interfaces and
worlds identified by `namespace:name@version`; a *world* is one component's
import/export contract; an *interface* is a named set of functions and types.

Options:

1. **Rename `Package` to `World`.** Matches the note's example and what the
   form most nearly produces (the file becomes one component with one world).
   But the argument it carries — `demo:app@0.1.0` — is *literally* a CM package
   ID, and worlds in WIT are named things *inside* packages, so this rename
   arguably makes the naming less CM-aligned, not more.
2. **Keep `Package`, add the missing vocabulary as it arrives.** The file
   declares a package (correct: the string is a package ID) that today contains
   one implicit world; goal 4 work that makes the full WIT surface expressible
   introduces `World`/`Interface` as their own named constructs when
   multi-interface/multi-world surfaces exist.
3. **Wholesale audit against an explicit rule.**

Recommendation: **adopt the rule from option 3, which mostly resolves to
option 2: every wavelet construct takes the name of the WIT construct it
lowers to.** Under that rule `Package` is already the right name — the rename
suggested in the original note appears to be backwards, and I'd keep it. The
real deliverable is a short audit of the remaining surface (`Import`/`Export`
keys, `elem:`/`as:` option names, error-message vocabulary, docs prose) against
the WIT glossary, with any renames done now while churn is cheap, and
`World`/`Interface` reserved as form names for goal 4 rather than spent here.
