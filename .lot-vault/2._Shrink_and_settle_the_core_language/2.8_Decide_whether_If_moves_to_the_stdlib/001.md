# 2.8 Decide whether If moves to the stdlib

Spun out of 2.4 (item 5), per Logan's update 003 there: the decision moves
`Do` from special form to bundled stdlib macro and keeps `Match` core,
because under total types the checker needs Match natively for exhaustiveness
checking and case narrowing (3.12). This note asks the follow-on question: if
Match stays, can `If` leave core too?

## Where the language stands today

`If c t e` is a special form in the interpreter (interp.rs `special_form`,
`if-MACRO`): it evaluates the condition, requires a bool, and jumps to
exactly one branch. Everything it does is expressible as a two-clause Match,
since patterns already include the bool literals `true`/`false`
(pattern-matching.mdx):

    If c t e   ⇒   Match c [(true t) (false e)]

The expansion preserves If's laziness (Match evaluates only the taken
clause's result) and its two-way branch shape.

## Question

Should `If` become a bundled stdlib macro expanding to `Match`, leaving
`Match` as the sole core branching construct?

## Options

1. **Move If to the stdlib as a macro over Match, sequenced after the
   checker work (recommended).** Core loses a special form; the interpreter
   and backend agree on If by construction, because there is only Match to
   agree on; and If's typing comes free from Match's — a `true`/`false`
   literal-clause pair over a bool scrutinee is exhaustive, and a non-bool
   condition becomes a checker error (today it is a runtime "condition must
   be a bool" error). Sequencing riders: it needs 3.6 (check macro-expanded
   code) so diagnostics on the expansion map back to the `If` the user
   wrote, and 3.12 (exhaustiveness) so the checker actually proves the
   two-clause expansion total. The backend must also compile bool-literal
   Match down to a plain wasm branch so codegen doesn't regress — worth
   verifying in emit.rs, since If is the hottest form in the examples.

2. **Keep If special.** Zero migration risk and the best diagnostics for
   free. But it leaves a form in core that is definable in userland, against
   the principle behind the Do decision — anything expressible as a bundled
   macro with no *additional* checker involvement moves out. If's checker
   involvement (bool condition, joining the branch types) is exactly Match's,
   not extra.

3. **Move If immediately, alongside Do.** Same endpoint as option 1 but
   before 3.6/3.12 exist, which would temporarily regress If's error
   messages to raw Match-expansion errors on every non-bool condition — a
   highly visible surface traded away for no schedule gain.

## Recommendation

**Option 1: yes, move If to the stdlib, but sequence it behind 3.6 and
3.12.** (`Do` doesn't need to wait; it has no checker involvement at all.)
Implementation slot: 5.7 (split the standard library from the core
language), same as Do. Until then If stays a special form. If 3.6/3.12 slip
badly, option 2 remains a respectable resting point — Match-in-core is the
load-bearing decision, and it is already made.
