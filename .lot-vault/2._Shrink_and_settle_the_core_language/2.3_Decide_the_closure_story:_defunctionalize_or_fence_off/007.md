**DECIDED (Logan, 2026-07-03): defunctionalization, with resource-lifting as
the boundary extension** — the recommendation in update 006, accepted as
stated. Concretely:

1. **Inside a component: defunctionalize (option 1).** Every `Fn` becomes a
   case of a generated per-arrow-type variant (discriminant = which function,
   payload = its concretely-typed captures); application is one generated
   dispatch function per arrow type. Closures are plain WIT variants; no
   headers, no private heap family.
2. **Cross-file combinators: expand-time (first cut).** Generic
   `map`/`filter`-style combinators are compile-time template expansions
   (5.7.1) imported via the existing macro-component machinery, so the
   combinator body lands in the calling component and local `Fn`s never
   escape. A function type in a genuinely cross-component signature is
   rejected with an explicit diagnostic (today it is unspellable by accident;
   the first cut makes that a deliberate "not yet").
3. **Boundary extension (designed, not gating anything): resource-lifting.**
   When a real cross-component callback need appears, an escaping closure
   lifts into a compiler-generated single-`call` resource whose
   implementation is the generated dispatch applied to the stored variant;
   inbound, a lone-`call` resource becomes one extra dispatch case holding
   the handle (a plain ABI value). Borrow-only first; `own` + drop rides on
   4.5's general resource work.

Consequences recorded across the plan:

- 5.8 (implementation) rescoped to items 1-2; new child 5.8.1 holds item 3.
- 2.2.7 (contingent drop of first-class functions) is moot — noted there.
- 2.3.1's question is answered by this decision — noted there, Thing closable.
- 5.1's contingency is resolved: zero non-ABI residue, so option 1

- 5.7.1 is now load-bearing for item 2 — noted there, with 7.4 (an import

- 8.2 (closures over the wire) stays parked but remains feasible only via

  standalone call-component could embed. Resource handles cannot serve 8.2.
