This is the options-and-recommendation write-up requested in 2.3.1. Decision
only; implementation is 5.8, and you review the concrete design before any of
it lands.

**Question: do closures compile by defunctionalization — every `Fn` in a
component becomes a case of a generated WIT variant, applied through a
generated dispatch function — or does the component keep one private, non-ABI
closure representation fenced inside itself?**

Options:

1. **Defunctionalize (recommended).** Per arrow type, the compiler emits a
   variant (one case per function of that type; discriminant = which function,
   payload = its captures, concretely typed) plus one dispatch function.
   - What it buys: closures ARE plain WIT variants — ABI-native values with no
     headers, no private heap family. They can cross component boundaries and
     be serialized, which settles "Closures over the wire" (8.2) as a follow-on
     rather than a design effort. It leaves *zero* non-ABI residue, which lets
     5.1 take the simplest memory story (headerless arenas) without carve-outs.
   - What it costs: apply sites compile to a branch over the variant (mitigable:
     a single-case variant devirtualizes to a direct call); the set of functions
     per arrow type must be closed per component (it is — compilation is
     whole-file); one generated variant + dispatcher per arrow type in the
     component (code size, negligible at current scale).
   - Design risks to settle in the 5.8 review: a *deterministic naming scheme*
     for the generated variants so WIT output is stable across edits; and what
     happens when a function type appears in an *exported* signature — sensible
     initial cut is to reject that and lift the restriction later, since the
     generated variant would otherwise become public API.
2. **Private non-ABI representation, confined inside the component.** The goal
   weakens to "ABI except functions": function values can never cross a
   boundary or be serialized, and 5.1 must keep headers (or a wasm-GC family)
   for this one kind of value — a permanent asterisk on ABI-native, and the
   sole reason headers would still exist anywhere.
3. **Fallback: drop first-class functions** (2.2.7) — top-level function
   references only, or hand-written variant-plus-dispatch.

Recommendation: **option 1.** The closed-world requirement — normally the
dealbreaker for defunctionalization — is free here because compilation is
already whole-file, and every cost it does have is local and bounded, while
option 2's cost is global and permanent (headers exist, the memory story forks,
and "ABI-native" carries an exception clause forever). Branchy dispatch is an
acceptable prototype cost with a known optimization path.
