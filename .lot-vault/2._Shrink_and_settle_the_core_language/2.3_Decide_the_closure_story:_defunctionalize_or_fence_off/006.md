Correction to my previous update: I leaned on docs/docs/components.mdx as design
authority. Checked against the implementation and the vault, that was wrong in
three places. Re-grounded analysis below; the revised recommendation is at the
end (it moves — the first cut gets smaller).

**What the docs got wrong or oversold:**

- The "Closures across boundaries" section (auto-lift into a single-`call`
  resource, inbound duck-typing) is aspirational prose. Nothing in src/
  implements any of it, and no vault Thing records it as decided — 2.3 (this
  Thing) IS the decision point, and its recorded options don't include it. So
  resource-lifting enters this conversation as a fresh proposal on its merits,
  not as "the documented design" as I framed it.
- The docs' resource status ("one concrete resource — cell") undersells the
  implementation: the backend also handles imported opaque host resources and
  the set functor's *compiler-generated, component-implemented* exported
  resource (wit.rs emits the `resource set {...}` interface from the same
  op-table that drives inference). General user-declared resources are 4.5,
  in progress, scoped to the conformance counter.
- `compose --fuse` (my cited recovery path for per-element boundary-call cost)
  exists only in docs prose. No vault Thing schedules it; don't weight it.

**What the implementation actually says:**

- Closures today are `TAG_FN` boxes in the uniform boxed heap — [tag,
  table-slot, n-captures, capture boxes], applied by `call_indirect`
  (emit.rs). Classic closure conversion, exactly the representation goal 5
  replaces, and the thing that "never could cross a boundary" per this
  Thing's opening note.
- An arrow type in a boundary signature is not "rejected" — it is
  *unspellable*. Signature types map to concrete WIT types only; there is no
  syntax for a function-typed param in an Export/Import. The boundary gap
  can't even be written down today.
- `map`/`filter`/`fold`/`zip` are interpreter **builtins** (builtins.rs) —
  they don't live in any file, so "import map from another file" is currently
  a non-feature, not a broken feature.

**Point 2 revised — the vault already answers it, at expand time.** The
planned home for combinators is 5.7 (split stdlib from core) + 5.7.1
(monomorphize the higher-order stdlib): map/filter/fold/zip become
compile-time template expansions — "macros that expand to typed
tail-recursive loops" — specialized per element type at each call site. And
the machinery for importing compile-time code across files already exists and
ships: build.rs compiles a `DefMacro` file into a `wavelet:meta/macros`
component and resolves macro imports at expand time. So "file A imports a
combinator from file B and calls it with a local Fn" is, per the vault's own
plan, an *expand-time* import: the combinator's body lands inside the calling
component, the local Fn never escapes, and per-component defunctionalization
covers it with nothing extra. (Ergonomic dependency to note: 7.4 — an import
can't yet be both a runtime dependency and a macro library, which combinator
libraries will want.) My earlier generic-vs-monomorphic split stands, but the
generic branch isn't merely "a way this could work" — it's the already-planned
5.7.1 path.

**Point 3 revised — resources are viable and cheaper than I thought, but
they're a proposal, not precedent.** The set functor is the exact pattern a
closure-lift needs: a compiler-generated resource interface implemented by
the emitting component. 4.5 is building general resource machinery anyway, so
a lifted `resource fn-string-to-string { call: ... }` rides on scheduled work
rather than adding a new family. The composition with defunctionalization is
unchanged (the resource's `call` = the generated dispatch applied to the
stored variant; inbound handles become one extra dispatch case holding a
plain i32 handle — still headerless, 5.1 option 1 survives). Caveats revised:
own-handles need 4.5's drop story (borrow-only is the cheap first cut), and
the n-boundary-calls-per-map cost now has *no* scheduled mitigation, which is
a real reason to prefer the expand-time path for combinators and reserve
lifting for genuine cross-component callbacks (handlers, subscriptions).

**8.2 corrected — it's more ambitious than "serialize the variant", and it
cuts for defunctionalization.** The vault's 8.2 is Erlang-shaped: serialize a
closure as a small *standalone component* that non-wavelet components can
invoke. Resource handles can't do this at all (per-instance identity). A
defunctionalized closure can, as a follow-on: its captures are concretely
typed data, so the compiler can emit a component exporting `call` = the
dispatch branch for that case over an embedded payload. Option 2 (private
representation) forecloses 8.2 entirely. That asymmetry is additional weight
for option 1 independent of anything else in this update.

**Revised recommendation.** Option 1 (defunctionalize inside the component)
unchanged, and now with one more argument (8.2). The cross-component question
your point 2 raised is a separable rider with two cuts:

1. *First cut (recommended):* combinators are expand-time (5.7.1 — already
   the plan), and a function type in a genuinely cross-component signature
   stays unspellable, now as an explicit "not yet" rather than a syntax
   accident. This satisfies "import map, call it with my local Fn" with zero
   boundary machinery, because the import happens at expand time.
2. *Extension when a real cross-component callback appears:* resource-lifting
   (borrow-only first), building on 4.5's machinery. Keep it in the design
   drawer as the intended shape, but don't gate 2.3 or 5.8 on it.
