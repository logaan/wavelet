Continuing the conversation — responses to the three points, grounded in what the
repo already says (docs/docs/components.mdx, docs/docs/language/type-system.mdx,
docs/docs/roadmap.mdx).

**1. Agreed** — and note that in Wavelet one *file is one component*
(components.mdx), so "never escapes the file" and "never escapes the component"
are the same condition. File-local is exactly where defunctionalization's closed
world lives.

**2. Cross-file combinators split into two cases, and the split does the work.**
The type system is monomorphic — nothing generic survives into WIT
(type-system.mdx). So:

- A *generic* `map`/`filter` (usable at any element type) cannot be a component
  export at all: its WIT signature is unwritable. Generic combinators have to
  ship the way generics already ship — as macro/functor libraries, expanded at
  compile time *into the calling component*. Once expanded, the closure never
  escapes, and defunctionalization covers it with zero extra machinery. The
  everyday "import map, call it with a local lambda" experience is satisfied
  this way, provided stdlib combinators are macro-level.
- A *monomorphic* combinator (`each: func(f: ???, items: list<string>) ->
  list<string>`) genuinely crosses a component boundary, and here
  defunctionalization alone genuinely cannot serve: the caller's generated
  variant is caller-specific (its case set is the caller's function set, its
  dispatch function lives in the caller), so the callee's independently-compiled
  signature can't name it. Point 2 does force a second mechanism.

**3. Resources — yes, and it's already the documented interop design.**
components.mdx §"Closures across boundaries" sketches precisely this: an
escaping closure lifts automatically into a single-method resource —

    resource fn-string-to-string { call: func(arg: string) -> string; }
    each: func(f: borrow<fn-string-to-string>, items: list<string>) -> list<string>;

— with inbound duck-typing of any lone-`call` resource. The roadmap lists it as
designed-but-not-implemented. The nice part is that it *composes* with option 1
rather than competing:

- Inside the component, a closure is the defunctionalized variant.
- When one escapes, the compiler stores the variant in a caller-owned resource
  entry and passes the handle; the resource's `call` implementation IS the
  generated dispatch function applied to the stored variant. The internal
  representation never leaks.
- Inbound, a lone-`call` resource ducks in as a function value: the arrow
  type's dispatch variant gains one extra case holding the foreign handle,
  whose branch calls through it. A handle is a plain ABI value (an i32 into a
  host-managed table), so even this case is headerless — 5.1's simplest memory
  story survives intact.
- It also settles update 003's "function type in an exported signature" design
  risk better than the proposed reject-initially cut: the signature lifts to
  the resource form instead of being rejected.

Costs and limits to keep in view:

- **8.2 stays parked.** A handle is per-instance identity, not a serializable
  value, so "closures over the wire" does *not* come free via resources. (The
  defunctionalized variant itself is serializable, but only meaningful to a
  component that knows the case set — fine, 8.2 was parked anyway.)
- **Lifetimes.** `borrow<>` scopes the handle to the call — the right default
  for combinators, which never keep the function. A callee that *retains* a
  closure (callback registration) needs an `own` handle plus the general
  resource drop story, which is the same not-yet-implemented roadmap item —
  the two features land together naturally. A borrow-only first cut is cheap
  and covers map/filter/each.
- **Performance.** Mapping over n items costs n boundary calls, each copying
  args per the canonical ABI. That's inherent to cross-component callbacks, not
  to this representation; the planned `compose --fuse` is the recovery path for
  all-Wavelet graphs (merge modules, devirtualize `call` back into direct
  dispatch).
- **Other WIT types considered:** a plain exported variant fails because the
  case set is caller-specific and dispatch lives in the caller; a bare function
  index has no meaning across instances; instantiation-time function imports
  are static wiring, not first-class values. Resources are the only WIT-native
  carrier of "callable owned by someone else."

One implementation-side note for whenever 5.8 happens: the interpreter/runner
currently passes closures across simulated component boundaries freely (imports
resolve by package id inside one heap). Since interp.rs is the semantics oracle,
the boundary rules (lifting, borrow scoping, duck-typing) need to be modelled
there — or flagged in check.rs — before emit grows them, or the two will
disagree on exactly the programs this decision is about.

**Where this leaves the decision** (still not locking in): the three points
converge on a two-level story — defunctionalize *inside* the component (option 1
stands), lift escaping closures to single-`call` resources *at* the boundary
(already the documented design), and distribute generic combinators as
macro/functor libraries so they expand into the caller and never escape. That
keeps "closures ARE plain WIT types" true everywhere: variants inside, handles
at the edges.
